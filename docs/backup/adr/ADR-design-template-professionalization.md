# ADR: 设计模板专业化

## 状态

Proposed

## 背景

当前五类模板已经覆盖业务、方案、实现、测试和集成设计，但主要是标题和简短注释。它们不能稳定约束：

- evidence / decision / assumption / risk / requirement ID
- 4+1 和 C4 图示
- 质量属性场景
- 接口契约和数据模型
- 风险驱动测试
- 跨阶段一致性和 traceability
- 从设计产物派生开发任务和测试用例

## 决策

升级设计模板体系：

1. 所有 artifact 增加 frontmatter。
2. 所有章节说明“写什么、质量标准、证据要求、图示要求、常见错误、是否必填、下游用途、对应 Gate”。
3. business-design 引入业务规则、决策表、状态模型、验收标准、需求追溯矩阵。
4. solution-design 引入 C4、4+1、接口契约、数据流、质量属性场景、STRIDE、ADR。
5. implementation-design 引入模块/文件/函数设计、调用链、状态机、事务/并发/幂等、日志监控、测试钩子。
6. test-design 引入测试金字塔、风险驱动测试、覆盖矩阵、不可测项、转测准入。
7. integrated-design 引入 artifact version/hash、跨阶段一致性、风险汇总、可开发/可测试/可发布结论。

## 替代方案

### 方案 A：保持当前轻模板

优点：填写成本低。

缺点：质量不稳定，无法 gate。

### 方案 B：引入完整 arc42/企业模板

优点：成熟完整。

缺点：过重，不适合 feature 级任务。

### 方案 C：只依赖 Review 补缺口

优点：初始产物简单。

缺点：返工多，review 成本高。

## 取舍

选择专业化但可裁剪的模板。每个章节可允许“不适用，理由：...”，但关键 trace 和风险项不得留空。

## 后果

正面：

- 输出更专业。
- 支持质量门禁。
- 支持从设计到开发/测试派生。

负面：

- 模板更长。
- 小任务需要明确裁剪规则。
- 需要 doc-lint 和示例 fixtures 防止模板退化。
