# Feature Design 中 Evidence/Decision 维护规则放置方式调研

## 1. 调研问题与范围

本次只比较 `feature-design` 内的两种实现方式：

- **方案 A：全局规则约束**——在 `SKILL.md` 中集中定义 Evidence/Decision 的维护原则，要求 Agent 在整个设计流程中遵守。
- **方案 B：任务内动作**——把初始化、登记和核对动作嵌入实际发生这些语义事件的任务步骤，并给出可检查的完成条件。

外部 Skill 不在本次比较范围内。调研只使用 Anthropic/Agent Skills 官方资料和原始 Skill 仓库资料。资料没有直接规定“Evidence/Decision 应放在哪个标题下”，因此本文将**来源可直接支持的事实**与**结合当前仓库得出的设计推论**分开陈述。

## 2. 当前 Feature Design 的事实基线

当前 [`skills/feature-design/SKILL.md`](../../skills/feature-design/SKILL.md) 定义五个线性顶层任务：

1. 恢复工作空间并加载专业上下文；
2. 完成并确认核心设计；
3. 形成 Draft；
4. 隔离 Review 并修订；
5. 批准并发布 Baseline。

现有 Skill 只在任务 1、任务 2 中把 Evidence/Decision 当作已有输入读取，没有定义下列维护行为：

- 当前设计活动的 Decision 文档何时初始化或恢复；
- `knowledge-query` 候选被主会话采用后何时登记 Evidence；
- 用户确认实质取舍后何时登记 Decision；
- Review 引入新事实或改变实质取舍后如何补充记录；
- 写入失败时如何揭示，而不把知识记录变成 Draft、Review 或发布门禁。

仓库已有的权威约束是：

- 只有实际支持或改变设计的查询结果才登记 Evidence；
- Decision 只记录实质取舍；
- Evidence/Decision 用于保存知识，不控制对话、Draft、Review 或发布；
- 查询 Subagent 只返回候选，只有主会话能够采用并登记 Evidence。

因此，问题不是缺少数据模型，而是 Skill 没有把既有数据模型转化为可执行动作。

## 3. 一手资料发现

### 3.1 Anthropic：复杂关键操作应拆成清晰步骤并跟踪

Anthropic 的 [Skill authoring best practices](https://platform.claude.com/docs/en/agents-and-tools/agent-skills/best-practices) 明确建议：

- 复杂操作应拆成清晰、连续的步骤；
- 特别复杂的流程应提供可复制并逐项勾选的 Checklist；
- 清晰步骤能够防止 Claude 跳过关键验证，Checklist 用于跟踪多步骤流程进度；
- 质量关键操作应形成“验证、修复、重新验证”的反馈循环，并在要求满足后才继续。

同一指南还要求根据任务的脆弱性和变化程度选择自由度：当一致性关键、操作易错或必须遵循特定顺序时，应采用更具体的低自由度指令；当判断依赖上下文且存在多种有效做法时，才适合高自由度启发式指令。

**资料能支持的事实**：关键且易遗漏的动作，仅作为一般性原则出现，不如把它们放入明确步骤、跟踪项和反馈循环更符合官方推荐模式。

**资料不能证明的内容**：官方资料没有声称“只要写成步骤就绝不会遗漏”，也没有给出全局规则与局部步骤的定量对照实验。

### 3.2 Anthropic：核心流程与按需参考应分层

同一官方指南将 `SKILL.md` 定位为概览和导航入口，详细材料可以渐进披露到外部文件；只有在需要时才加载对应 Reference。它同时提醒应观察 Claude 是否遗漏重要引用、忽略内容或过度依赖某个部分，再基于真实行为调整结构。

[Agent Skills Specification](https://agentskills.io/specification) 规定 `SKILL.md` 是必需的 metadata + instructions 入口，`references/` 和 `scripts/` 是可选资源；规范本身没有定义 Markdown 各章节之间的执行优先级，也没有“全局规则会自动在每一步重新触发”的运行时机制。

**资料能支持的事实**：必须影响所有运行分支的核心行为应保留在 `SKILL.md`；条件性专业细节可以按需披露。仅把规则放进 `SKILL.md` 并不等于它获得了额外的确定性执行机制。

### 3.3 Anthropic：Skill 是否可靠必须靠真实评估验证

官方最佳实践把 evaluations 称为衡量 Skill 有效性的事实来源，建议先记录无 Skill 时的具体失败，再写最小指令并迭代测试。官方还给出实际观察例子：某条规则虽然已经写进 Skill，Claude 仍可能遗漏；改进方式可能是提高规则显著性、改用更强措辞，或者重构流程章节，然后再次测试。

**资料能支持的事实**：不能只通过文档静态推断两种方案谁一定可靠；最终必须以覆盖新建、恢复、知识采用、用户决策和 Review 修订的实际会话评估验证。

### 3.4 `obra/superpowers`：从“描述流程”转向“任务化执行”

当前 [`brainstorming/SKILL.md`](https://github.com/obra/superpowers/blob/main/skills/brainstorming/SKILL.md) 同时使用：

- 全局硬约束，例如用户批准设计前禁止进入实现；
- 强制 Checklist，要求为各项创建任务并按顺序完成；
- 与 Checklist 对应的具体 Process；
- 写完 Spec 后的自检和用户批准门。

该仓库的 [release notes](https://github.com/obra/superpowers/blob/main/RELEASE-NOTES.md) 记录了引入这些结构的原因：模型曾跳过设计阶段，或把整个讨论压缩成一段文本；作者随后把 Skill 从“描述工作流”改为通过 hard gate、mandatory checklist 和 process flow 强化执行。

这是一份原始实现及维护者经验记录，而不是受控实验，但它提供了与本问题直接相关的现实信号：**仅描述跨流程意图容易被跳过；把关键行为变成任务和门更容易强化遵循。**

## 4. 两种方案对比

| 维度 | 方案 A：全局规则约束 | 方案 B：任务内动作 |
|---|---|---|
| 语义单一来源 | 强。Evidence/Decision 的定义、边界和非门禁原则可集中维护 | 若在多个任务重复完整规则，容易漂移；需要避免复制语义 |
| 动作触发时机 | 弱。Agent 需要自己把“持续维护”映射到知识采用、用户确认、Review 修订等事件 | 强。可直接与“采用候选”“确认取舍”“修订设计”等事件共置 |
| 可检查性 | 弱。很难从“遵守全局规则”判断当前动作是否已执行 | 强。可以检查登记成功，或写入失败已明确揭示 |
| 抗遗漏能力 | 较弱。属于跨越长流程的记忆性约束 | 较强。任务清单和局部步骤会在临近事件时再次出现 |
| 对自然设计主线的干扰 | 小，但容易成为无实际行为的旁注 | 控制得当时小；若新增独立顶层任务或频繁核对，会挤占语义分析主线 |
| 维护成本 | 低，单点修改 | 若复制规则则高；若只放短触发动作则可控 |
| 非门禁要求 | 容易表达“不控制 Draft/Review/发布”，但也因此难保证实际登记 | 需要明确区分“必须尝试并揭示结果”和“写入成功才能继续”，否则会误造新门禁 |
| 与当前五任务结构的匹配 | 只能提供横切原则，不能填补各任务缺失动作 | 可自然落到任务 1、2、4，无需增加第六个顶层任务 |

## 5. 具体结论

### 5.1 若必须二选一，应选择任务内动作

Evidence/Decision 维护不是供 Agent 偶尔查阅的领域知识，而是有明确时点的副作用：恢复时初始化/读取，候选知识被采用时登记 Evidence，实质取舍确认时登记 Decision，Review 改变语义时补充记录。

因此，单纯采用方案 A 的问题不是规则不正确，而是缺少：

- **何时执行**；
- **紧邻哪个语义事件执行**；
- **怎样判断已执行或失败已揭示**。

根据官方“复杂流程用清晰步骤和 Checklist 防止跳步”的建议，以及 `superpowers` 从描述流程转向任务化执行的实际经验，方案 B 对行为的约束更可靠。

### 5.2 推荐采用“任务内动作作为执行机制，集中语义作为单一来源”

这不是另建第三套流程，也不是把相同规则重复到多处。具体信息分层应为：

- 在 `SKILL.md` 用一个很短的 **Evidence/Decision 语义不变量**集中定义“记录什么、不记录什么、由谁记录、不得成为门禁”；它只作为单一语义来源。
- 在任务 1、2、4 中只写**局部触发动作和可观察结果**，不重复完整定义；它们才是维护行为的执行机制。

换言之，选择方案 B 负责“做”，保留最小全局定义只负责“含义”。这符合 `writing-great-skills` 的信息层级：步骤是 Agent 当前要执行的主层，规则和定义是次级 Reference；也符合单一来源和共置原则。

## 6. 对当前 Skill 的落位建议

### 6.1 任务 1：恢复与初始化

在当前 `init-design` 之后：

- 初始化或读取当前 `<designType>` 的 Decision 文档；
- 读取当前设计实际采用的 Evidence/Decision；
- 将其纳入恢复后的设计模型。

可观察结果：当前活动的 Decision 容器存在且可读取，既有记录已恢复。初始化失败必须明确揭示，但不能阻断 Draft、Review 或发布。

### 6.2 任务 2：在语义事件发生时立即登记

在语义分析循环内增加两个局部触发：

- 主会话明确采用 `knowledge-query` 候选，且该知识实际支持或改变设计时，立即登记一条多来源 Evidence；未采用的候选和 `gap` 不登记。
- 用户确认一个实质取舍后，立即登记 Decision，并引用实际支撑该决定的 EV ID；普通事实、临时理解、开放问题和无实质替代方案的细节不登记。

任务 2 的完成检查不应要求“所有写入都成功”才能进入 Draft，而应检查：所有已触发的维护动作都已执行；若工具失败，失败及未持久化内容已向用户明确揭示。这样强化维护动作，但不新增知识记录门禁。

### 6.3 任务 4：只处理 Review 引入的新语义事件

Review 本身不自动产生 Evidence/Decision。只有当主会话在修订过程中：

- 采用了新的外部/项目知识，才登记新 Evidence；
- 与用户重新确认了新的实质取舍，才登记新 Decision。

不应在 Review 结束时机械补录所有 finding，也不应把普通修订包装成 Decision。

### 6.4 不建议新增独立顶层维护任务

Evidence/Decision 是设计活动中的横切知识副产物，不是独立设计阶段。新增第六个顶层任务会延迟登记、破坏“在语义事件发生时记录”的共置关系，也会让维护动作抢占核心设计主线。

## 7. 实现前仍需明确的边界

当前 Decision CLI 只有 `init/read/add`，是追加模型。若 Review 推翻了既有 Decision，现有接口没有显式的 `supersedes`、修订或作废语义。Skill 不应自行发明更新行为；在具体重构前，需要决定是：

- 追加一个新 Decision，并在现有字段中说明其影响；还是
- 扩展 Decision 数据模型表达替代关系。

这不影响“任务内触发比纯全局规则更可靠”的结论，但会影响任务 4 能写多具体。

## 8. 建议的验证场景

根据 Anthropic 的 evaluation-driven 建议，至少用以下真实会话验证重构，而不是只做 Markdown 静态检查：

1. 新建设计：无外部知识查询，但产生一个实质用户取舍；只应生成 Decision。
2. 知识采用：查询返回多来源候选，主会话只采用其中一部分；只登记被采用且影响设计的 Evidence。
3. 查询 gap：没有答案时继续设计，不登记 Evidence。
4. 恢复会话：已有 Evidence/Decision 能被恢复，不重复初始化或重复登记。
5. Review 修订：普通措辞修复不登记；采用新事实或重新确认取舍时才补充记录。
6. 写入失败：Agent 明确揭示持久化失败，但仍能按既有合同形成 Draft、Review 和发布。

评估重点应是实际行为是否发生、是否过度记录、是否误造成门禁，而不只是 Skill 中是否出现了 Evidence/Decision 关键词。
