# Solution Design Guide 与 Spec 优化方案

## 文档状态

- 状态：已确认，已实施
- 适用范围：`feature-design` 中 `solutionDesign` 对应的 Design Guide 与 Spec
- 实施方式：评审确认后，由 `writing-great-skills` 原则指导具体重构
- 上位约束：遵循 `06-skill-first-feature-design-refactor.md`，不改变既有 Skill-first 架构

## 1. 背景与问题

当前 `feature-design` 已经形成稳定的通用执行过程：恢复工作空间、识别 Design Type、加载专业 Reference、执行 design tree/frontier 语义分析、形成 Draft、隔离 Review、人工批准、发布 Baseline 并同步状态。

现有 Solution Design Guide 和 Spec 没有破坏这一过程，但专业表达过于概括：

- Guide 只列出通用架构原则和若干分析工具，未充分定义方案设计的目标、粒度和必须关闭的系统级不确定性；
- 4+1、C4、ATAM、STRIDE 被并列为可选透镜，无法稳定支撑特性级完整架构认知；
- Spec 将系统边界、架构、接口、数据、质量属性和风险压缩在少量抽象章节中，追溯和适用性结构明显，可读的方案叙事不足；
- 新建特性与存量增强缺少不同的专业分析规则，Agent 容易把增量方案写成局部补丁；
- 可靠性、安全隐私韧性和一般非功能质量属性缺少独立可见的设计位置。

本次优化不增加流程步骤，而是增强 `solutionDesign` 分支加载后的专业判断力和 Draft 内容合同。

## 2. 目标

### 2.1 专业目标

方案设计以特性整体为主语，形成一套人类可理解、工程上可执行、结果可验证、关键结论可追溯的系统级目标态方案。

方案必须确定：

1. 系统责任边界；
2. 架构与微服务职责；
3. 接口与集成契约；
4. 数据归属与一致性；
5. 关键质量属性；
6. 异常与失败语义。

完成后的方案应使后续实现、测试和评审可以直接消费上述系统级结论，无须重新猜测具有全局影响或高返工代价的问题，同时保留微服务内部实现结构的设计空间。

### 2.2 Skill 目标

- 保持主 Skill 的五项 Task 和语义分析循环为唯一执行过程；
- 让 Solution Design Guide 只提供方案设计专业 Reference；
- 让 Solution Design Spec 只定义 Draft 内容和表达合同；
- 通过清晰的上下文指针，在识别为 `solutionDesign` 后一次性加载 Guide 与 Spec；
- 固定专业检查维度，动态缩放分析深度、对话顺序和文档篇幅；
- 使新建特性和存量增强共用同一 Guide 与 Spec。

## 3. 范围边界

### 3.1 本次修改的专业语义

专业语义只修改：

- `skills/feature-design/references/design-guides/solution-design.md`
- `skills/feature-design/references/specs/solution-design.md`

### 3.2 必要的兼容性接缝

现有 Lint 在 `scripts/devsphere-design.js` 中硬编码了 Solution Design 的六个旧标题和集中式适用性标签。新 Spec 若采用十四章和五个 4+1 子章节，必须同步调整以下确定性接缝：

- `solutionDesign.coreSections`；
- 4+1 子章节存在性检查；
- 固定专项章节的适用性结论检查；
- 对应测试夹具和合同断言。

这些改动只让确定性校验接受新的 Spec，不引入新的专业判断、状态或流程行为。

### 3.3 保持不变

- `feature-design/SKILL.md` 的五项顶层 Task；
- 调查事实、design tree/frontier、推荐与挑战、单个高价值问题、重新计算和动态分段确认；
- Design Type 的独立性，不增加固定顺序或上游 Baseline 门禁；
- Work、Draft、Lint、Review、Approval、Artifact 和状态同步语义；
- 单个隔离 `design-reviewer` 串行执行全部适用 Checklist 的 Review 机制；
- 现有 Review Checklist 的评审规则和检查项；
- `requiredDesignTypes` 外层策略；
- `knowledge-query`、Evidence 与 Decision 边界；
- 跨任务的长期设计知识库、唯一特性文档定位和自动合并机制。

## 4. 信息层级与单一事实源

依据 `writing-great-skills` 的信息层级：

| 内容 | 唯一事实源 | 原因 |
|---|---|---|
| 五项执行任务和语义分析循环 | `feature-design/SKILL.md` | 每次设计活动都需要，属于主步骤 |
| 方案设计目标、边界、专业原则、4+1、强制决策和收敛标准 | Solution Design Guide | 只在 `solutionDesign` 分支需要，属于渐进加载的专业 Reference |
| Draft 章节、内容职责和表达规则 | Solution Design Spec | 只在形成和核对 Solution Design Draft 时使用 |
| 专项评审规则和具体检查项 | 现有 Review Checklists | Review 时按适用性加载，保持独立单一事实源 |
| 标题、非空内容和适用性格式 | 确定性 Lint | 只判断机器可确认的结构事实 |

Guide 不复制主 Skill 的步骤，Spec 不包含对话流程，Checklist 不复制 Spec 章节说明。

## 5. Solution Design Guide 重构方案

Guide 是一份平级专业 Reference，不是一套顺序执行的子 Skill。Agent 在主 Skill 的语义分析循环中持续使用这些规则建立 design tree、选择 frontier 和判断收敛。

Guide 实施时保留既有 Design Guide 的公共标题合同，并在其中增加必要的专业章节。目标标题为：

```markdown
## 专业边界
## 设计场景
## 专业原则
## 分析透镜
## 必须关闭的系统级决策
## 高价值矛盾
## 风险缩放
## Checklist 导航
## 专业收敛标准
```

### 5.1 专业边界

Guide 使用“目标态”作为方案设计的核心概念：

> 目标态是本次设计生效后，特性在系统中的完整有效架构。它完整到足以独立理解系统级行为和契约，但不穷举微服务内部实现细节。

应明确：

- 设计主语是特性整体，而不是单个模块或一次局部代码修改；
- 方案设计确定特性涉及哪些系统和微服务、各自负责什么、如何协作；
- 开发视图可以到微服务和服务级关键组件，具体类、方法、表字段、SQL 和逐文件步骤由实现设计继续细化；
- 相关需求、业务设计、现有方案、代码和运行事实按当前目标采用，不把任何特定 Design Type 的 Baseline 设为固定前置条件；
- 缺少会实质影响架构判断的事实时，明确缺口及影响，并在语义分析循环中调查或与用户确认。

### 5.2 新建与存量增强

新建和增量是 Guide 内部的专业分析分支，不成为新状态、命令参数或顶层 Task。

#### 新建特性

- 识别目标、场景、系统环境和既有约束；
- 从零建立特性目标态；
- 说明新增责任如何进入现有系统边界。

#### 存量特性增强

先基于现有正式设计、代码和运行事实建立可信现状，再识别：

- **新增**：本次引入的场景、职责、契约、数据、质量目标和风险；
- **受影响**：因新增能力必须调整或重新验证的存量内容；
- **保持不变**：仍然有效且本次必须保护的既有边界和契约；
- **非目标**：当前方案明确不承担的能力和演进范围。

设计讨论聚焦变化及其影响，最终 Draft 以完整目标态为正文，通过前置变更范围和轻量变化摘要提高本次评审效率。Draft 不采用必须拼接历史增量文档才能理解的补丁式表达。

### 5.3 专业原则

- 先用核心场景和总体方案建立连续认知，再展开架构视图和横切契约；
- 每项架构复杂度都对应明确目标、约束、质量属性或风险；
- 系统边界、微服务职责、依赖方向和数据所有权相互闭合；
- 视图章节解释方案如何成立，专项章节固化必须稳定遵循的契约；
- 同一决策只在一个位置完整定义，其他位置说明其作用并引用结论；
- 接口同时表达成功、业务失败、技术失败、部分成功、权限、幂等、超时、重试、兼容和演进语义；
- 质量属性使用刺激、环境、响应和可判定目标表达；
- 追溯关系证明覆盖完整，正文保持面向人类阅读的方案叙事；
- 低影响内容使用核验范围、结论和依据进行简洁表达，高影响内容展开到足以关闭系统级不确定性。

### 5.4 分析透镜：4+1 完整性框架

4+1 是 Solution Design 的默认完整性框架。C4、时序图、状态图、活动图、数据流图和部署图是表达各视图的工具。

#### 场景视图

作为阅读入口，使用代表性正常、异常和高风险边界场景说明触发者、参与系统、端到端过程和可观察结果。

#### 逻辑视图

说明能力分解、系统边界、微服务职责、行为和数据所有权以及依赖方向，回答“谁负责什么”。

#### 进程视图

说明同步和异步交互、状态流转、并发、超时、重试、幂等、补偿、降级及部分失败，回答“运行时如何保持正确”。

#### 开发视图

说明受影响微服务、服务级关键组件、公共能力、代码仓或构建单元以及依赖方向，回答“软件组织如何承载方案”。粒度止于微服务和服务级关键组件。

#### 物理视图

说明部署单元、中间件、存储、网络和信任边界、扩缩容、资源隔离、高可用、容灾及运维影响。拓扑不变时仍记录检查范围、容量结论和沿用依据。

五个视图都必须形成有效结论。固定的是观察维度，不是图形数量、讨论顺序或篇幅。

### 5.5 必须关闭的系统级决策

| 决策 | 必须形成的结论 | 主要表达位置 |
|---|---|---|
| 系统责任边界 | 本系统负责、不负责什么；外部边界如何协作 | 总体方案、逻辑视图 |
| 架构与微服务职责 | 参与微服务、责任分配、依赖方向、行为所有者 | 逻辑视图、开发视图 |
| 接口与集成契约 | 调用关系以及成功、失败、权限、幂等、超时、兼容语义 | 进程视图、接口与集成设计 |
| 数据归属与一致性 | 权威所有者、流向、生命周期、一致性和演进原则 | 逻辑视图、数据设计 |
| 关键质量属性 | 适用场景、可判定目标、架构措施和验证方式 | 物理视图、可靠性、安全、非功能章节 |
| 异常与失败语义 | 重复、超时、部分失败、依赖不可用、恢复和用户可见结果 | 场景视图、进程视图及对应专项章节 |

实现机制可以继续细化，表中系统级语义必须在 Solution Design 中形成明确结论。

### 5.6 高价值矛盾

- 新增能力突破存量边界，但责任和数据所有权没有同步调整；
- 总体方案描述了目标态，真实代码或运行事实仍依赖不同边界；
- 微服务承担责任却不拥有所需行为或数据；
- 正常路径成立，重复、并发、部分失败、版本混用或依赖不可用时语义失效；
- 场景、逻辑、进程、开发和物理视图描述的不是同一套方案；
- 接口契约把恢复责任同时交给调用方和提供方，或双方都未承担；
- 质量属性只有形容词，没有具体环境、响应和边界；
- 引入的架构复杂度没有对应目标、风险或可验证收益；
- 增量设计只解释新增组件，没有说明受影响和保持不变的既有契约；
- 推荐方案的不可逆成本和残余风险没有进入用户决策。

### 5.7 风险缩放

设计深度由架构影响决定，不由需求标题、新建/增强标签或模板篇幅决定。

以下因素提高展开深度：

- 跨系统、跨微服务或跨进程协作；
- 新的数据所有权、一致性边界或迁移要求；
- 信任边界、敏感数据、高危操作或外部输入；
- 容量、性能、可用性、功能安全或容灾目标；
- 外部依赖、共享中间件或多版本并存；
- 不可逆技术选择或高返工代价；
- 对存量契约、部署或运行资源的广泛影响。

局部低风险方案仍完成全部固定检查维度，但使用简洁、可核验的结论表达。

### 5.8 Checklist 导航

- 始终执行：`architecture-consistency`、`design-traceability`；
- 持久或共享数据的模型、所有权、生命周期或访问方式变化时执行：`data-review`；
- 跨进程、网络、队列、存储或可用性边界影响正确性时执行：`reliability-review`；
- 存在信任边界、身份、权限、秘密、外部输入、敏感数据或滥用路径时执行：`security-review`；
- 生产运行、诊断、发布、回滚、容量或值守责任受到影响时执行：`operability-review`。

章节固定存在不等于对应条件 Checklist 必然执行。主会话根据 Draft 的实际设计影响判断适用性，并沿用现有隔离 Review 机制。

### 5.9 专业收敛标准

同时满足以下条件，Guide 的专业覆盖才算完成：

- 核心正常、异常和高风险场景能够端到端解释；
- 五个 4+1 视图都已检查并形成有效结论；
- 六类系统级决策均有明确、相互一致的结论；
- 新建场景的新增责任，或增量场景的新增、受影响、保持不变和非目标已经明确；
- 跨微服务职责、接口、数据、失败和质量属性不需要后续人员重新猜测；
- 关键技术取舍、不可逆代价和残余风险已经确认；
- Draft 可以脱离聊天上下文独立阅读，并能突出方案重点；
- 所有固定章节都有实质设计内容，或包含核验范围、结论和依据。

该标准补充主 Skill 的通用收敛条件，不复制或替代主 Skill 的执行步骤。

## 6. Solution Design Spec 重构方案

### 6.1 Spec 定位

Spec 是 Draft 内容合同，不是提问清单、设计顺序或对话脚本。Agent 在主 Skill 中按 design tree/frontier 动态完成设计，收敛后才将 Confirmed Design 组织为以下固定结构。

Spec 文件实施时继续保留所有 Design Spec 共用的元结构：

```markdown
## 核心章节
## 适用性说明
## 条件章节
```

“核心章节”定义十四章 Draft 合同；“适用性说明”定义固定章节如何按影响缩放；“条件章节”定义出现特定风险时，应在既有固定章节中进一步展开哪些内容。三个元标题属于 Spec Reference，不写入最终 Draft。

### 6.2 Frontmatter

保持现有合同：

```yaml
---
artifactId: "SD-<TASK_ID>"
version: "<SEMVER>"
---
```

不增加 `designMode`、`baselineRefs`、内部状态或跨任务知识库字段。新建/增量及相关设计输入在正文中表达。

### 6.3 固定正文结构

Draft 使用以下顺序和标题：

```markdown
# Solution Design

## 概述
## 特性需求与设计上下文
## 总体方案
## 4+1 架构视图
### 场景视图
### 逻辑视图
### 进程视图
### 开发视图
### 物理视图
## 接口与集成设计
## 数据设计
## 可靠性、可用性与功能安全设计
## 安全、隐私与韧性设计
## 非功能质量属性设计
## 关键技术决策、取舍与风险
## 下游设计约束与交接
## 需求追溯与覆盖关系
## 词汇表
## 参考资料
```

十四个主章节和五个视图子章节固定存在。固定章节约束最终产物覆盖，不规定主会话的讨论顺序。

### 6.4 各章内容合同

#### 概述

- 文档目的、范围和目标读者；
- 特性在系统或子系统中的定位；
- 采用的设计输入和关键约束。

#### 特性需求与设计上下文

- 理解方案所需的业务背景、价值和关键需求摘要；
- 核心业务场景与当前系统能力；
- 新建或存量增强的设计场景；
- 增量场景的新增、受影响、保持不变和非目标；
- 直接和间接影响范围。

本章提供阅读上下文，不重新生产完整业务设计。

#### 总体方案

- 设计目标和现状问题；
- 总体设计思路和系统责任边界；
- 总体架构及核心功能实现原理；
- 推荐方案、关键依据和主要取舍；
- 增量场景的轻量变化摘要。

读者阅读本章后应能理解方案重点，再进入视图和专项契约。

#### 4+1 架构视图

五个子章节遵循 Guide 的视图职责，描述同一个目标态方案：

- 场景视图串联代表性正常、异常和高风险场景；
- 逻辑视图定义能力、边界、微服务职责和所有权；
- 进程视图定义运行时协作及失败状态；
- 开发视图定义微服务和服务级关键组件的承载关系；
- 物理视图定义部署、资源、网络、高可用和运维结构。

图形按表达需要使用。正文必须解释图中的关键判断、关系和约束。

#### 接口与集成设计

- 内部接口、外部接口、事件和消息；
- 调用方、提供方、责任边界和输入输出的业务语义；
- 成功、业务失败、技术失败和部分成功；
- 认证、授权、幂等、超时和重试责任；
- 兼容、版本演进及外部依赖不可用时的行为。

#### 数据设计

- 核心数据对象及关系；
- 权威所有者、创建修改权限和数据流向；
- 生命周期、一致性和事务边界；
- 缓存、复制、冗余和派生数据；
- 迁移、回填、兼容、回滚和修复原则；
- 敏感数据标识及处理约束。

默认停留在系统级逻辑数据设计。构成跨服务稳定契约的物理细节可以明确，其余物理表结构由实现设计继续细化。

#### 可靠性、可用性与功能安全设计

- 关键功能流和故障范围；
- 故障模式、影响、检测、隔离、告警和恢复；
- 单点、冗余、超时、重试、补偿和降级；
- 过载、资源保护、升级兼容和回退；
- 高危操作、人因差错和功能安全适用性；
- 可接受的残余风险。

#### 安全、隐私与韧性设计

- 资产、敏感数据、威胁主体和信任边界；
- 数据流、威胁建模、认证和授权；
- 外部输入、高危操作、秘密和数据保护；
- 审计、隐私风险和个人数据处理；
- 攻击下的隔离、降级、恢复和业务韧性；
- 可接受的残余安全风险。

#### 非功能质量属性设计

- 性能、容量和可扩展性；
- 可维护性、可测试性和可服务性；
- 可观测性、兼容性和演进能力；
- 其他影响方案成立的质量属性。

每项适用属性包含场景、刺激与环境、可判定目标、架构措施和验证方式。

#### 关键技术决策、取舍与风险

- 推荐方案和主要替代方案；
- 选择理由和关键代价；
- 不可逆或高成本决策；
- 架构风险、缓解措施和残余风险；
- 实际支持结论的重要 Evidence 和 Decision。

本章集中呈现影响方案成立和评审判断的关键决策，不汇总普通设计细节。

#### 下游设计约束与交接

按受影响微服务或消费对象说明：

- 需要继续细化的服务内部模块和实现机制；
- 必须遵循的接口、数据和失败语义；
- 质量、安全、可靠性和运维目标；
- 测试必须验证的核心场景和风险；
- 当前仍然成立的假设、限制和残余风险。

本章表达消费合同，不建立 Design Type 的固定顺序或依赖门禁。

#### 需求追溯与覆盖关系

- 关键需求与方案落点；
- 核心场景与架构视图；
- 质量目标与保障机制；
- 风险与验证活动；
- 需要后续设计或实施消费的结论。

追溯结构保持轻量，能够证明覆盖完整即可，不以高密度编号或映射矩阵组织正文。

#### 词汇表

- 业务术语；
- 系统、微服务和服务级组件名称；
- 核心数据对象；
- 缩写和容易产生歧义的概念。

#### 参考资料

- 实际采用的需求或相关设计；
- 现有系统、接口、数据和工程规范；
- 安全、可靠性和行业标准；
- 业界方案及其他 Evidence 来源。

只列实际支持设计判断的资料。

### 6.5 固定章节的适用性说明规则

固定章节承担强制检查维度，但展开深度动态缩放。

存在实质影响时，章节给出完整设计。没有变化或影响很低时，章节至少说明：

- 核验范围；
- 当前结论；
- 判断依据；
- 目标态为何可以沿用现状或无需增加机制。

使用上述可核验结论替代“无”“不适用”或泛化工程描述。可靠性、安全隐私韧性、非功能质量属性以及低影响的 4+1 视图均遵循此规则。

### 6.6 条件内容规则

条件内容在固定章节内部展开，不新增另一套 Draft 目录：

- 存在跨进程或外部依赖时，在进程视图、接口与集成设计、可靠性设计中展开超时、重试、幂等、降级和恢复；
- 存在持久或共享数据变化时，在逻辑视图和数据设计中展开所有权、生命周期、一致性、兼容、迁移和回滚；
- 存在信任边界、权限、敏感数据或外部输入时，在物理视图和安全隐私韧性设计中展开威胁、授权位置和保护措施；
- 存在部署、容量或运维影响时，在物理视图、可靠性设计和非功能质量属性设计中展开拓扑、资源隔离、发布、回滚和可观测目标。

### 6.7 增量设计表达规则

存量增强采用：

> 目标态正文 + 前置变更范围 + 轻量变化摘要

- “特性需求与设计上下文”集中说明新增、受影响、保持不变和非目标；
- “总体方案”简要说明现状、问题、扩展方式和架构级变化摘要；
- 后续视图和专项章节描述变化后的完整有效方案；
- 正文可使用“新增”“调整”“沿用”帮助评审定位关键变化；
- 正式契约表达目标态语义，不只记录本次改动动作。

本规则只约束当前 Draft 的可读性和完整性，不定义跨任务长期设计文档的自动合并方式。

## 7. Lint 兼容方案

Lint 继续只检查确定性事实，不评价专业质量。

### 7.1 主章节

将 `solutionDesign.coreSections` 调整为十四个固定主章节标题。每个章节必须存在且包含非空正文。

### 7.2 4+1 子章节

为 `solutionDesign` 增加五个必选子章节元数据。Lint 检查它们位于“4+1 架构视图”内并包含非空正文。

### 7.3 适用性

取消 Solution Design 对集中式 `## 适用性说明` 以及四行标签格式的依赖。

Lint 对下列章节检查是否包含实质正文：

- 可靠性、可用性与功能安全设计；
- 安全、隐私与韧性设计；
- 非功能质量属性设计；
- 五个 4+1 视图。

“结论是否专业成立”由主会话语义分析和适用 Checklist Review 判断，不在 Lint 中增加语义规则。

### 7.4 兼容边界

- 只调整 `solutionDesign` 元数据和必要的通用标题提取能力；
- 其他 Design Type 的现有核心章节和集中式适用性格式保持不变；
- 不增加模板渲染器、文档生成器或语义 Lint；
- Draft hash、Review、Approval、publish 和 reopen 逻辑保持不变。

## 8. Review 兼容方案

不修改现有 Checklist 内容和集中 Review 机制，只更新 Guide 中的 Checklist 导航。

Review 仍然遵循：

- 每轮冻结 Draft 对应一个新的隔离 `design-reviewer`；
- `design-reviewer` 只读取冻结 Draft、全部适用 Checklist 和必要正式事实，并串行执行；
- 主会话汇总 findings 并与用户讨论修订；
- 语义修改使全部适用 Review 失效；
- blocking findings 关闭后才能发布。

十四章结构用于提高 Review 定位和专业内容可见性，不产生 Review Matrix 或章节级 Reviewer 编排。

## 11. 验收标准

方案实施完成后必须同时满足：

- 主 Skill 的五项 Task、语义分析循环和状态维护未发生行为变化；
