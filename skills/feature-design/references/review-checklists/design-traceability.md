# 相关设计追溯评审

## 适用条件

所有设计类型必选。根据当前 Design Type 评审其声明采用的 Requirement Baseline、上游 Design Baseline、代码事实和正式约束；不存在某类上游输入时，检查独立边界和依据是否明确。

## 评审规则

- 从已批准目标、范围、验收、正式输入和授权后移事项反向追踪到 Draft，再从下游交接反查每项约束是否来自已确认设计。
- Draft 不得静默弱化、扩大、固定或矛盾其正式输入；追溯关系清楚即可，不强制表格、ID 密度或固定映射格式。
- 任何使用 ASCII 字符模拟的语义图都是 `blocking` finding；适合图示表达的内容采用 Mermaid 后才能通过。没有图示本身不构成 finding，也不要求固定图示数量。
- `blocking`：目标无设计落点；正式范围或排除项被突破；新增未批准的结果责任；上游语义被改写；后移事项越过最迟决策点；下游交接来自未确认假设。
- `advisory`：只用于不影响追溯成立的可读性或低风险定位改进。
- `risk`：追溯已经明确，但仍存在已揭示的输入限制、假设或残余影响。

## 检查项

- [ ] Requirement 目标和核心验收是否都有当前设计落点？
- [ ] 正式范围、重要排除项和结果责任是否被完整遵守，没有静默扩张或弱化？
- [ ] Business Design 的验收是否只是对 Requirement Acceptance 的业务判据细化，而非另立需求成功口径？
- [ ] Requirement Baseline 中未固化的方案偏好是否保持为设计空间，而非被错误固定为约束？
- [ ] 用户授权后移事项是否在记录的最迟决策点（第一个依赖它的设计内容冻结前）关闭，或仍处于允许后移范围？
- [ ] 关键规则、接口、状态、风险或约束是否准确消费相关正式输入，没有遗漏或改义？
- [ ] 新增假设、实质取舍和输入限制是否显式表达并具有可核验依据？
- [ ] 相关 Artifact 的版本、语义和适用边界是否被正确引用？
- [ ] 下游交接中的每项强制约束是否来自当前 Draft 已确认的设计，而非新增未确认方案？
- [ ] 缺少相关输入时，独立使用边界、事实缺口及其实际影响是否明确？
- [ ] 流程、类关系、调用时序、活动、状态、组件或同类语义关系需要图示时，是否统一使用 Mermaid 且没有 ASCII 语义图？界面设计、视觉稿、Markdown 表格、目录树和代码片段是否被正确识别为非强制或非语义图内容？

## Reviewer 边界

执行本 Checklist 时，对 Draft 和正式 Artifact 保持只读；findings 纳入本轮统一 Review 结果，不与用户交互，不替用户决定需求或设计取舍。评审不增加 Requirement Baseline 未承诺的能力，也不把某一 Design Type 的专业内容强加给其他类型。
