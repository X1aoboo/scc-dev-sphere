# 业务设计指南

## 专业边界

Business Design 基于已经明确的需求语义，建立 Feature 生效后的**业务语义目标态**：一套完整、一致、可验证且技术实现中立的业务规则体系与业务行为定义。它关闭核心需求定义之外、但会影响业务结果一致性和下游设计的业务不确定性。

可以定义业务概念，对象、事件和状态的业务含义，角色和责任，业务规则与业务时间，成功、失败、部分成功和异常结果，以及业务级准确性、时效、权限、可见性和留痕约束。系统、微服务、接口、消息、数据库、表字段、中间件、计算引擎，以及技术重试、事务、补偿和一致性机制由 Solution Design 决定。删除技术实现描述后，业务定义仍须完整成立。

### 阶段边界判断

- **返回需求澄清**：不同答案会改变需求目标、Feature 核心业务含义、范围承诺、核心结果责任或 Requirement Baseline 的验收结论。
- **留在业务设计**：在核心需求语义确定后，补全对象适用范围、规则组合、时间归属、状态变化、异常结果或场景级判据。
- **留给方案设计**：业务结果已经确定，只剩系统责任、取数或计算方式、接口和消息、数据存储、一致性、重试、恢复、性能、安全或可靠性机制。

Requirement Baseline 定义为什么做、为谁做、做什么、不做什么和什么结果表示需求成功；Business Design 细化业务概念、规则、时间、状态、结果与场景级验收，保持其核心语义不变；Solution Design 消费已确认的业务语义并选择技术实现。

## 设计场景

### 新建特性

从 Requirement Baseline 建立新的完整业务语义目标态。

### 存量增强

先依据已有正式设计、业务规范、代码行为和运行事实建立可信现状，再识别新增、受影响、保持不变和非目标。讨论聚焦变化，最终 Business Design 描述变化生效后的完整目标态，使下游无需拼接新旧材料才能理解当前业务语义。

## 专业原则

- 业务语义优先于流程表现；流程、旅程、状态图、规则表和计算示例只是按需表达工具。
- Requirement Baseline 的核心语义保持不变；发现必须改变目标、范围或验收时返回需求澄清。
- 同一概念具有唯一含义，相同业务事实得到相同业务结果。
- 每条规则具有明确适用范围；规则组合、优先级和冲突处理形成闭合判定。
- 时间、状态、场景和验收使用同一口径；场景实例化规则，不成为隐藏规则的唯一位置。
- 同一结论只在一个权威位置完整定义，其他位置引用或说明作用。
- 技术约束不得反向改写业务语义。
- 设计深度随语义影响缩放；“沿用现状”以核验依据为前提。

图示不是必选产物，不要求每份设计必须包含图示，也不规定图示数量。仅当业务流程、参与者协作、状态或生命周期关系使用图示能明显降低理解成本时才绘制，并统一使用 Mermaid；禁止使用 ASCII 字符模拟语义图。界面设计、视觉稿等不适合 Mermaid 的内容不强制绘图。

## 业务语义分析透镜

以下六个维度是 design tree/frontier 的固定完整性透镜，不是问卷、执行顺序或必选图表：

1. **业务概念与度量语义**：概念、对象、事件、分类、状态、度量及其口径是否唯一。
2. **适用范围、参与者与业务责任**：规则适用于谁、什么对象和范围，由谁发起、确认、修正并承担结果责任。
3. **业务规则与判定逻辑**：条件、默认行为、组合、优先级和冲突是否能推出唯一结果。
4. **时间、状态与生命周期语义**：时间基准、周期、归属、状态、版本和历史结果是否闭合。
5. **业务场景、异常与结果语义**：正常、失败、部分成功和关键边界下最终发生什么、参与者观察到什么。
6. **存量影响与业务验收契约**：新旧承诺、目标态完整性、Requirement Acceptance 的细化和下游验证语义是否一致。

## 必须关闭的业务级决策

### 业务概念与度量语义

覆盖适用的核心业务概念、对象、事件、分类和状态含义，对象关系，指标、金额、数量、比例、公式变量，以及单位、精度、舍入和空值语义。核心公式或需求身份属于 Requirement Baseline 时，只补全其完整业务语义，不自行改写。

### 适用范围、参与者与业务责任

明确适用角色和对象，组织、租户、地域、产品或数据范围，触发、生效和排除条件，发起、确认、撤销和修正权，以及业务结果责任方。

### 业务规则与判定逻辑

覆盖适用的纳入、排除、计算、分类、去重、聚合、默认行为、优先级、冲突处理、特殊对象和无法判定时的业务行为。完成标准是每组关键条件都能推出唯一业务结果。

### 时间、状态与生命周期语义

覆盖适用的业务时间基准、统计周期、归属时间、生效、失效、截止、过期、状态及转换、跨周期、迟到、补录、回溯、规则版本、历史重算和结果冻结。

### 业务场景、异常与结果语义

覆盖适用的正常成功、整体失败、部分成功、空结果、重复、冲突、取消、撤销、前置条件变化、对象失效、数据缺失或冲突、暂不可判定、人工修正，以及参与者可观察结果。

### 存量影响与业务验收契约

明确新增、受影响、保持不变和非目标，新旧规则关系，历史对象和结果影响，Requirement Acceptance 如何细化为场景与规则判据，存量承诺如何保护，以及授权后移事项是否到达最迟决策点。

## 高价值矛盾

- 业务规则静默改变 Requirement Baseline，或新规则静默扩大需求范围。
- 同一概念存在多套口径，或相同事实可能得到不同结果。
- 多规则组合冲突、优先级不闭合，或规则缺少适用范围。
- 正常路径完整但异常结果缺失，或业务结果可计算却不可解释。
- 时间、状态、版本、回溯和冻结规则互相冲突。
- 场景结果无法由权威规则推出，或验收使用另一套业务定义。
- 存量增强只写新增内容，破坏既有承诺或留下不完整目标态。
- 业务定义依赖具体技术实现，技术限制反向决定业务含义。

## 风险缩放

设计深度由业务语义影响决定，不由需求名称或类型决定。概念口径、规则组合、跨周期结果、不可逆责任、历史影响和存量承诺的风险越高，越需要展开规则、场景、状态或示例。

低影响或沿用现状的内容可以用“核验范围 + 当前结论 + 判断依据 + 沿用理由”简洁表达。所有维度仍需形成可核验结论，但不要求复杂流程图、状态机、规则矩阵或领域模型。

## 专业收敛标准

- Requirement Baseline 未被静默修改，最迟决策点均已处理。
- 核心业务概念具有唯一解释，适用范围和业务责任明确。
- 关键规则可以唯一判定，时间、状态和生命周期语义闭合。
- 正常、异常和关键边界结果明确，场景结果可由规则推出。
- 业务验收使用与正文相同的业务语义。
- 存量增强形成完整目标态并保护仍有效的业务承诺。
- 业务定义技术实现中立。
- Solution Design 无需重新决定关键业务语义。
