# 方案设计指南

## 专业边界

以特性整体为主语设计**目标态**：本次设计生效后，特性在系统中的完整有效架构。目标态应让读者独立理解系统级行为和契约，同时保留微服务内部实现结构的设计空间。

确定特性涉及哪些系统和微服务、各自承担什么职责以及如何协作。开发视图可以到微服务和服务级关键组件；类、方法、表字段、SQL 和逐文件步骤由更细粒度的设计继续展开。

按当前目标采用相关需求、业务设计、既有方案、代码和运行事实。任何特定 Design Type 的 Baseline 都不是固定前置条件；缺少会改变架构判断的事实时，明确缺口和影响。

## 设计场景

### 新建特性

从目标、核心场景、系统环境和既有约束建立目标态，说明新增责任如何进入现有系统边界。

### 存量特性增强

先从正式设计、代码和运行事实建立可信现状，再识别：

- **新增**：本次引入的场景、职责、契约、数据、质量目标和风险；
- **受影响**：必须调整或重新验证的存量内容；
- **保持不变**：继续有效且必须保护的既有边界和契约；
- **非目标**：当前方案明确不承担的能力和演进范围。

设计推演聚焦变化及其影响，最终方案描述完整目标态，并用前置变更范围和轻量变化摘要帮助评审定位本次变化。

## 专业原则

- 先用核心场景和总体方案建立连续认知，再展开架构视图和横切契约。
- 每项架构复杂度都服务于明确目标、约束、质量属性或风险。
- 系统边界、微服务职责、依赖方向和数据所有权相互闭合。
- 视图解释方案如何成立，专项设计固化必须稳定遵循的契约。
- 同一决策只在一个位置完整定义，其他位置说明其作用并引用结论。
- 接口同时表达成功、业务失败、技术失败、部分成功、权限、幂等、超时、重试、兼容和演进语义。
- 质量属性使用刺激、环境、响应和可判定目标表达。
- 正文保持面向人类阅读的方案叙事，追溯关系用于证明覆盖完整。
- 低影响内容使用核验范围、结论和依据简洁表达；高影响内容展开到足以关闭系统级不确定性。

## 分析透镜

以 4+1 作为默认完整性框架，五个视图都形成有效结论：

- **场景视图**：用代表性正常、异常和高风险边界场景串联触发者、参与系统、端到端过程与可观察结果。
- **逻辑视图**：说明能力分解、系统边界、微服务职责、行为和数据所有权以及依赖方向。
- **进程视图**：说明同步和异步交互、状态流转、并发、超时、重试、幂等、补偿、降级及部分失败。
- **开发视图**：说明受影响微服务、服务级关键组件、公共能力、代码仓或构建单元以及依赖方向。
- **物理视图**：说明部署单元、中间件、存储、网络和信任边界、扩缩容、资源隔离、高可用、容灾及运维影响；拓扑不变时记录核验范围、容量结论和沿用依据。

C4、时序图、状态图、活动图、数据流图和部署图是表达视图的工具。图示不是必选产物，不要求每份设计必须包含图示，也不规定图示数量。仅当系统关系、协作时序、状态、活动、数据流或部署关系使用图示能明显降低理解成本时才绘制，并统一使用 Mermaid；禁止使用 ASCII 字符模拟语义图。界面设计、视觉稿等不适合 Mermaid 的内容不强制绘图。

## 必须关闭的系统级决策

1. **系统责任边界**：本系统负责和不负责什么，外部边界如何协作。
2. **架构与微服务职责**：参与微服务、责任分配、依赖方向、行为所有者和数据所有者。
3. **接口与集成契约**：调用关系以及成功、失败、权限、幂等、超时、重试、兼容和演进语义。
4. **数据归属与一致性**：权威来源、流向、生命周期、一致性边界以及迁移和回滚原则。
5. **关键质量属性**：适用场景、可判定目标、架构措施、验证方式和实现代价。
6. **异常与失败语义**：重复、并发、超时、部分失败、依赖不可用、恢复以及用户可见结果。

具体实现机制可以继续细化，上述系统级语义必须形成明确且相互一致的结论。

## 高价值矛盾

- 新增能力突破存量边界，但责任和数据所有权没有同步调整。
- 目标态与真实代码或运行事实依赖不同边界。
- 微服务承担责任却不拥有所需行为或数据。
- 正常路径成立，重复、并发、部分失败、版本混用或依赖不可用时语义失效。
- 五个 4+1 视图描述的不是同一套方案。
- 接口恢复责任重叠或无人承担。
- 质量属性只有形容词，没有具体环境、响应和边界。
- 架构复杂度没有对应目标、风险或可验证收益。
- 增量方案只解释新增组件，没有说明受影响和保持不变的既有契约。
- 不可逆成本和残余风险没有进入用户决策。

## 风险缩放

设计深度取决于架构影响，而不是需求标题、新建或增强标签。跨系统协作、数据所有权变化、信任边界、容量与可用性目标、外部依赖、多版本并存、不可逆选择和广泛存量影响越多，越需要完整展开视图、失败语义、质量属性和替代方案。

局部低风险方案仍检查全部固定维度，以简洁、可核验的结论表达。

## Checklist 导航

- 始终执行：[architecture-consistency](../review-checklists/architecture-consistency.md)、[design-traceability](../review-checklists/design-traceability.md)。
- 持久或共享数据的模型、所有权、生命周期或访问方式变化时执行：[data-review](../review-checklists/data-review.md)。
- 跨进程、网络、队列、存储或可用性边界影响正确性时执行：[reliability-review](../review-checklists/reliability-review.md)。
- 存在信任边界、身份、权限、秘密、外部输入、敏感数据或滥用路径时执行：[security-review](../review-checklists/security-review.md)。
- 生产运行、诊断、发布、回滚、容量或值守责任受到影响时执行：[operability-review](../review-checklists/operability-review.md)。

固定章节不自动触发条件 Checklist；按实际设计影响判断适用性。

## 专业收敛标准

- 核心正常、异常和高风险场景能够端到端解释。
- 五个 4+1 视图都已检查并形成有效结论。
- 六类系统级决策均有明确且相互一致的结论。
- 新建场景的新增责任，或增量场景的新增、受影响、保持不变和非目标已经明确。
- 跨微服务职责、接口、数据、失败和质量属性无须后续人员重新猜测。
- 关键技术取舍、不可逆代价和残余风险已经确认。
- 方案可脱离聊天上下文独立阅读，并能突出设计重点。
- 每个固定章节都有实质内容，或包含核验范围、结论和依据。
