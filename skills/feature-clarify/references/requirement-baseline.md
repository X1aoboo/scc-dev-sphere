# Requirement Baseline 语义契约

用本契约形成可供后续业务设计独立理解的 Requirement Draft，并在 Review 通过和用户批准后原样发布为 Requirement Baseline。

## 必须清楚表达的语义

### 问题与目标

- 原始问题、受影响对象及其可观察影响。
- 经过验证的需求目标，以及目标为何能够改善原始问题。
- 必要时区分期望结果与用户最初提出的候选解决手段。

### 需求边界

- 本次对哪些对象、场景、结果和既有能力变化承担责任。
- 容易被默认纳入、会造成明显范围膨胀或存在不同理解的重要排除项。
- 与目标和验收直接相关的已确认约束。

### 验收基线

- 每个核心目标对应的可观察或可判定结果。
- 足以消除多义性的参照对象、条件或业务状态。
- 核心目标和易误判边界所需的失败判定。
- 验收不得超出边界，也不得提前指定业务方案、技术实现或测试步骤。

### 用户授权后移事项

对每项后移内容表达：尚未决定的问题、可能影响的目标/边界/验收、当前建议与风险、用户已授权后移，以及第一个依赖它的设计内容冻结前这一最迟决策点。没有后移事项时直接说明没有。

## 区分约束与偏好

- 把用户已经确认、后续不得自行改变的内容写成正式需求约束。
- 把用户提出但尚未确认必须采用的流程、界面、技术或实现方向写成“未固化的方案偏好”，不要让它限制后续设计空间。
- 不要把 Agent 推断、查询 gap 或未授权遗漏写成需求事实。

## 表达自由度

根据需求复杂度选择最短的清晰结构。不要强制章节编号、固定表格或逐项填写不适用内容；不要扩写成完整 PRD，不加入用户旅程、详细业务流程、规则矩阵、接口、数据模型或技术方案。

判断质量的唯一标准是：后续业务设计无需依赖历史聊天重新解释需求，并能围绕同一目标、边界和验收开展工作。
